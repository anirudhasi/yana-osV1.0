Section 1

Executive Summary

What Yana OS is, who uses it, and the problems it solves.

Yana OS is a production-grade, microservices-based operations platform for managing a fleet of electric vehicles and the riders who operate them. It provides a complete digital backbone for EV-based gig workforce companies — from recruiting and verifying riders, to assigning vehicles, scheduling demand work shifts, collecting daily rent, training riders, and resolving support issues.

🏍️
Rider Lifecycle

End-to-end digital onboarding with KYC verification, document management, and status state machine from application to activation.

🚗
Fleet Management

Track every vehicle — GPS telemetry, battery health, allotments, maintenance schedules, and hub utilization.

💰
Payments & Wallet

Immutable double-entry ledger, automated daily rent collection, Razorpay/UPI autopay, incentives, and penalties.

📋
Demand Marketplace

Client demand slot publishing, smart rider matching, attendance check-in/out with GPS, and automated earnings calculation.

🎓
Skills & Training

Multilingual video training modules with quizzes, badges, points, streaks, and global leaderboard gamification.

🎧
Support

Full ticket lifecycle with SLA tracking, multi-channel messaging, WhatsApp integration, and satisfaction ratings.

Section 2

System Architecture

How all components connect, route, and communicate with each other.

Yana OS follows a microservices architecture where each domain is an independent Django service. All traffic is routed through an Nginx API Gateway. Asynchronous tasks are handled by Celery workers backed by Redis. File storage uses MinIO (S3-compatible). All services share a single PostgreSQL 15 database with separate schemas.

10-digit Indian mobile number. Tapping "Send OTP" calls POST /api/v1/auth/rider/send-otp. The auth service auto-creates a rider record if the phone is new.

2

Backend generates a 6-digit OTP. In development, OTP is logged to server console. In production, sent via SMS. Testing OTP: 123456.

RIDER

SYSTEM

6.3 — Data Flow: Allotment → Rent → Payment

This shows the financial data chain triggered by a vehicle allotment:

FLOW 
1. HUB_OPS creates allotment → VehicleAllotment created (status=ACTIVE)
   VehicleAllotment { rider_id, vehicle_id, daily_rent_amount=150, security_deposit=2000 }

2. Celery task fires → generates RentSchedule entries for 30 days
   RentSchedule × 30 rows { allotment_id, due_date, amount=150, status=PENDING }

3. Also fires → WalletLedger entry for security deposit
   WalletLedger { payment_type=SECURITY_DEPOSIT, amount=2000, direction=D }

4. Nightly Celery beat → processes due rent entries
   For each RentSchedule WHERE due_date=today AND status=PENDING:
     → Attempt UPI autopay via Razorpay API
     → On success:
         WalletLedger { payment_type=DAILY_RENT, amount=150, direction=D, balance_after=X }
         PaymentTransaction { status=SUCCESS, gateway_txn_id=pay_xxx }
         RentSchedule.status = PAID
     → On failure:
         RentSchedule.status = OVERDUE, days_overdue++, penalty applied

5. Vehicle returned → WalletLedger deposit refund entry
   WalletLedger { payment_type=DEPOSIT_REFUND, amount=2000, direction=C }

Section 7

Business Workflows

End-to-end operational playbooks for every key business process.

7.1 — Rider Onboarding Workflow

The onboarding workflow takes a new applicant from phone registration to an ACTIVE rider eligible for vehicle allotment and demand work. It spans 4 services.

1
RIDER
Phone Registration & OTP Login

Rider enters phone number in the app. Auth service sends OTP. Rider verifies → JWT tokens issued. Rider status: APPLIED

2
RIDER
Profile Completion

Rider fills name, email, DOB, gender, address, bank details via the app. Admin can also complete profile. Status: APPLIED

3
RIDER
Document Upload

Upload Aadhaar (front + back), PAN, Driving License (front + back), bank passbook/cancelled cheque, profile photo. Documents stored in MinIO. Status: DOCS_SUBMITTED → KYC_PENDING

4
ADMIN
Document Review & KYC Decision

SUPPORT_AGENT reviews each document. Can approve/reject individual documents with reason. Final KYC decision: VERIFIED or REJECTED. A rejected KYC logs the reason and notifies the rider. Status: UNDER_REVIEW → KYC_VERIFIED or KYC_FAILED

5
RIDER
Nominee Registration

Rider adds at least one nominee (family member) with relationship, phone, and Aadhaar. Required for insurance and emergency contact purposes.

6
RIDER
Mandatory Training

Rider completes all mandatory skill modules in the app (road safety, EV care, delivery best practices). Must pass quizzes. Skills service marks onboarding_readiness=True. Status: TRAINING

7
ADMIN
Rider Activation

HUB_OPS or CITY_ADMIN verifies the rider is physically at the hub. Calls POST /api/v1/riders/{id}/activate/. Rider status: ACTIVE. Rider is now eligible for vehicle allotment and demand slots.

8
SYSTEM
Wallet Created

On activation, Payments service auto-creates a RiderWallet with zero balance. Rider can now receive earnings and is subject to daily rent collection.

7.2 — Vehicle Allotment & Return Workflow

1
HUB_OPS
Select Vehicle & Rider

HUB_OPS navigates to Fleet → Allotments → New Allotment. Selects an AVAILABLE vehicle and an ACTIVE rider without a current allotment. Enters odometer, battery %, condition notes, daily rent, and security deposit amount.

2
SYSTEM
Allotment Created — Status Changes

VehicleAllotment created (status=ACTIVE). Vehicle status changed to ALLOCATED. Unique constraints enforced — cannot allot a vehicle already allocated or allot two vehicles to one rider simultaneously.

3
CELERY
Financial Setup

Celery task generates 30-day RentSchedule entries. Security deposit debited from rider's wallet (or invoiced if wallet insufficient). Nightly collection begins from next calendar day.

4
RIDER
Rider Operates Vehicle

Rider uses the EV for daily deliveries. GPS telemetry is ingested by the fleet-telemetry FastAPI sidecar (real-time). Battery level, location, and odometer updated continuously.

5
HUB_OPS
Vehicle Return

Calls POST /api/v1/fleet/allotments/{id}/return/. Records: odometer at return, battery %, condition at return, damage notes. Calculates km_driven = odometer_return − odometer_allotment. Allotment status: RETURNED. Vehicle status: AVAILABLE.

6
SYSTEM
Deposit Refund & Rent Schedule Closure

If no damage noted: security deposit refunded to wallet (WalletLedger: DEPOSIT_REFUND, Credit). If damage: deduct damage cost from deposit, refund remainder. Future RentSchedule entries marked WAIVED. Collection stops.

7.3 — Demand Slot & Earnings Workflow

1
SALES
Create Demand Slot

Sales team receives demand from client (e.g., Zepto needs 20 riders for tomorrow 6am–2pm at Koramangala). Creates slot: client, dark store, shift date/time, riders_required, pay_structure.

2
CITY_ADMIN
Publish Slot

Admin reviews the slot and publishes it. Status changes DRAFT → PUBLISHED. Slot is now visible in the rider app to eligible riders (filtered by city, hub, reliability score, badges).

3
RIDER
Apply for Slot

Rider views available jobs in the app, taps the Zepto slot card, reviews details (pay, timing, location), and taps "Apply". DemandApplication created with status=APPLIED. Matching engine calculates match_score.

4
CITY_ADMIN
Bulk Confirm Riders

Admin views applications sorted by match_score. Selects top applicants and bulk-confirms them. Application status: CONFIRMED. Slot riders_confirmed incremented. When riders_confirmed = riders_required, slot status: FILLED.

5
RIDER
Check-In at Dark Store

Day of shift: Rider arrives at Zepto dark store, taps Check-In in app. App records timestamp and GPS coordinates. These are validated against the dark store location to prevent proxy check-ins.

6
RIDER
Complete Shift & Check-Out

Rider completes deliveries (orders_completed tracked). At shift end, taps Check-Out. Application status: COMPLETED. hours_worked calculated. orders_completed finalized.

7
CELERY
Earnings Calculation & Payout

Post-shift Celery task calculates earnings: PER_ORDER slots: 15 orders × ₹50 = ₹750. PER_SHIFT: flat ₹500. Credits WalletLedger (payment_type=INCENTIVE, direction=C). Updates earnings_credited=True in application.

7.4 — Support Ticket Workflow

1
RIDER
Create Ticket

Rider selects category (Vehicle Issue, Payment Issue, etc.), enters subject and description. App calls POST /api/v1/support/tickets/. Ticket created with status=OPEN, priority=MEDIUM. Auto-assigned ticket number (TKT-YYYY-NNNNN).

2
SUPPORT_AGENT
Ticket Assignment

Agent sees new ticket in queue, assigns to themselves. Status: ASSIGNED. SLA timer starts. High/Critical priority tickets have shorter SLA windows (2h/4h vs 24h/48h for Low/Medium).

3
SUPPORT_AGENT
Investigation & Communication

Agent reviews ticket with linked entities (vehicle, allotment, payment). Posts messages visible to rider. Can add internal notes (is_internal=True) for team context. Status: IN_PROGRESS.

4
SUPPORT_AGENT
Resolution

Agent resolves the issue (issues refund, arranges vehicle swap, clarifies KYC). Posts resolution notes. Calls POST /api/v1/support/tickets/{id}/resolve/. Status: RESOLVED.

5
RIDER
Rating & Closure

Rider receives resolution notification, reviews, and rates satisfaction (1–5 stars). Ticket status: CLOSED. If unsatisfied, rider can escalate → Status: ESCALATED → Manager queue.

7.5 — Daily Rent Collection Workflow

1
CELERY BEAT (Nightly 00:01 IST)
Daily Rent Collection Job Fires

Celery beat scheduler triggers the collect_daily_rents task. Queries all RentSchedule entries where due_date=today AND status=PENDING.

2
SYSTEM
Check Wallet Balance & UPI Mandate

For each due entry: check if rider has active UPI mandate (razorpay_mandate_id). If mandate exists and balance sufficient → trigger Razorpay debit. If no mandate → create manual payment request.

3
RAZORPAY WEBHOOK
Payment Gateway Callback

Razorpay webhook fires on payment success/failure at POST /api/v1/payments/webhooks/razorpay/. Signature verified. On success: WalletLedger entry created, RentSchedule.status=PAID. On failure: status=OVERDUE.

4
SYSTEM
Overdue Handling & Penalties

Overdue entries accumulate days_overdue. After 3 days: penalty applied (overdue_penalty added). After 7 days: rider flagged for suspension review. After 14 days: force-return process initiated. All actions audit-logged.
